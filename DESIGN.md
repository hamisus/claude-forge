# Claude Forge: System Design

## Philosophy

Every software project starts the same way: someone has an idea, opens a text editor, and begins building. But before they can write meaningful code, they must answer dozens of questions: Where should this live? How do we structure it? What's the testing strategy? How do we organize work? What tools do we need? How will we debug?

With Claude Code, we've discovered that these questions aren't random. They follow predictable patterns. A React project needs different tooling than a Rust backend, but both follow the same logical workflow: understand the problem deeply before coding, build with intention, verify what you built, and capture what you learned so the next feature goes faster.

Claude Forge exists to automate the "setup tax" of starting new projects. The insight is simple: **80% of optimal Claude Code workflow is stack-agnostic, and only 20% is specific to your technology choices.** The universal 80% is the plan-build-verify-learn cycle. The stack-specific 20% is the tooling, patterns, and conventions that make that cycle efficient for your particular technology.

Claude Forge packages both into a reusable bootstrap system. The goal is moving from "I have an idea" to "I have a fully configured Claude Code development environment ready for serious work" in a single session, with enough scaffolding to sustain productive development for months.

## System Architecture

Claude Forge operates in two distinct phases:

### Phase A: Project Genesis (Runs Once)

Project genesis happens exactly once, at the start of a project's life. It's the "architecture intake" — the conversation that senior engineers usually have with clients, but automated and streamlined for speed.

**`/scope`** — Interactive Discovery Conversation

This command initiates a structured conversation designed to capture everything needed to bootstrap the project. It's not a form; it's a dialogue. The flow unfolds in four phases:

1. **Vision**: What are we building? Why does it exist? Who uses it? What problem does it solve?
2. **Technical Context**: Where does this run? What's the deployment model? What data does it work with? What systems integrate with it?
3. **Constraints**: How much time do we have? What's the team's experience level? Are there tooling preferences or organizational standards to follow?
4. **MVP Definition**: What's in version 1? What stays out? What's the absolute minimum that proves value?

The output is a structured `scope.json` file that captures all of this in a form that machines can parse but humans can still read and edit. This becomes the source of truth for the next two commands.

**`/spec`** — Specification Generation

Given the scope, `/spec` generates three foundational documents:

- `PROJECT_SPEC.md` — The human-readable statement of what we're building: objectives, success criteria, non-goals, and constraints.
- `ARCHITECTURE.md` — The technical blueprint: layers, key components, data flow, external dependencies, and architectural decisions with rationale.
- `PHASES.md` — The work breakdown: what features ship in each phase, in what order, and why.

These documents aren't generated by simple templates. They're reasoned artifacts. The system reads the scope, applies domain knowledge (recognizing patterns like "this is a mobile app," "this is a data pipeline," etc.), and produces architecture that's appropriate for the specific problem space. A document-processing backend gets different architecture guidance than a real-time multiplayer game.

**`/bootstrap`** — Stack-Specific Scaffolding

This is where generic architecture becomes concrete tooling. `/bootstrap` reads the spec and architecture, identifies the tech stack and project patterns, and generates:

- **Custom Skills**: Stack-specific workflows like `/new-component` for React projects, `/new-screen` for Flutter, or `/new-model` for Django. Each skill encodes the "right way" to add that type of artifact in this project.
- **Hooks**: Automated validators and code generators. A JavaScript project gets `eslint` hooks; a Rust project gets `clippy`. A Flutter project might auto-generate boilerplate for state management. These hooks run on every edit, keeping code quality consistent without friction.
- **Rules**: Project conventions encoded as guardrails. "We organize screens by feature, not by type." "All async operations must include error handling." "State changes are logged." These rules guide both AI and human developers.
- **Agents**: Specialized AI workers for stack-specific tasks. A code reviewer that understands your project's patterns. A test suggester that knows which testing library you use. These work alongside the developer, not instead of them.
- **CLAUDE.md**: The root configuration file that ties everything together. This is the human-readable handoff document that explains "here's what we built, here's how it works, here's what to do next."

The key insight: **generation is template-guided but AI-authored.** It's not filling in blanks or running string substitution. It's reasoning about what the project needs and creating appropriate artifacts. A Go microservice and a Django web app both need configuration management, but the implementation looks completely different, and `/bootstrap` knows that.

### Phase B: Development Loop (Runs Every Session)

Once genesis is complete, developers enter the normal development cycle. This is where the universal 80% dominates.

**`/dev <feature>`** — The Plan-Build-Verify Loop

This is the daily workhorse. It takes a feature description and orchestrates the full cycle:

1. **Plan**: Break down the feature. Identify what files need to change. Propose the approach. Pause for feedback.
2. **Build**: Implement the feature using the generated skills (/new-component, /new-test, etc.) and stack-aware hooks.
3. **Verify**: Run tests, check linting, validate against architecture rules, review the output for quality.

All of this happens in one command, but it's not linear. `/dev` pauses at natural checkpoints so the developer can steer mid-execution. Building a complex feature shouldn't feel like handing it off to a black box; it should feel collaborative.

**`/wrap-up`** — Session End Ritual

When a session ends, `/wrap-up` runs automatically (or manually). It:

- Summarizes what was accomplished
- Writes a human-readable handoff document for the next developer (or the same developer in a future session)
- Captures machine-readable context (which files changed, which tests passed, what we learned)
- Flags anything left incomplete or risky
- Suggests what to tackle next

This ensures that context doesn't evaporate at session boundaries. The next session starts with a clear picture of the current state, not a blank slate.

**`/tdd`** — Test-Driven Development Cycle

For projects that prioritize testing, `/tdd` orchestrates test-first development. It generates test stubs, verifies they fail correctly, and guides implementation. It's the testing-first variant of `/dev`.

**`/debug`** — Structured Debugging

When something's broken, `/debug` brings system thinking to bear. It captures the symptom, traces it to likely causes using stack-specific knowledge, proposes fixes, and validates them. It's faster and more thorough than ad-hoc debugging because it knows the codebase structure and common failure patterns for the stack.

**`/learn`** — Lesson Capture with Deduplication

Discoveries made during development don't disappear. `/learn` captures them: "We found that X approach is inefficient for our dataset size" or "This library has a footgun with async initialization." These lessons feed into a project knowledge base and update CLAUDE.md so future developers avoid the same pitfalls.

**`/phase-review`** — Phase Audit with Quality Scoring

At the end of each development phase, `/phase-review` audits the work. It checks:

- Did we deliver what was planned?
- Is the code quality acceptable?
- Are tests adequate?
- Does the architecture still make sense?
- Are there technical debts we should address?

It produces a quality score (5 dimensions × 20 points = 100 possible). Nothing ships below 80/100. If something's below that, it identifies specific gaps and suggests fixes.

**`/skill-check`** — Meta-Skill for Skill Health

As the project evolves, custom skills can become stale or misaligned with reality. `/skill-check` audits all generated skills, tests them, verifies they still reflect current patterns, and flags any that need updating. It's the system's way of staying sharp.

## The Three-Layer Skill Architecture

Claude Forge uses a three-layer skill model:

**Layer 1: Universal Skills**

These work for any project. `/dev`, `/wrap-up`, `/tdd`, `/debug`, `/learn`, `/phase-review`, and `/skill-check` are truly stack-agnostic. They exist in the template and work for Python projects, Rust projects, frontend projects, data projects — everything. They implement the plan-build-verify-learn cycle that works everywhere.

**Layer 2: Generated Skills**

When `/bootstrap` runs, it generates skills specific to the technology stack. A React project gets `/new-component`, `/new-page`, and `/refactor-component`. A backend service gets `/new-endpoint`, `/new-migration`, and `/add-middleware`. These skills are generated from templates but tailored to the specific patterns the `/scope` and `/spec` defined.

**Layer 3: Evolved Skills**

Over time, patterns emerge in development that aren't covered by generated skills. Maybe the project consistently needs to add new integrations. Maybe there's a complex testing pattern that keeps appearing. `/skill-check` detects these patterns and flags them to `/learn`, which can propose new skills. These evolved skills capture "the way we do this project" and make it faster and more consistent.

## Hook System Design

Hooks are the automation layer. They intercept operations and run stack-specific logic without interrupting the developer.

**Universal Hooks**

- **Compact Matcher Hook**: When context compaction happens, this hook re-injects essential project context (architecture summary, key conventions, current phase goals) so developers continue with a clear picture.
- **Notification Hook**: Desktop alerts for important events (test failures, phase reviews complete, etc.). Keeps developers aware without breaking focus.
- **Stop Hook**: End-of-session verification. Prompts to commit changes, run final tests, update CLAUDE.md before stopping.

**Generated Hooks**

When `/bootstrap` runs, it creates hooks specific to your stack:

- **PostToolUse on Edit/Write**: After editing code, a linter runs. After writing a React component, TypeScript and ESLint validate it. After writing Python, mypy and Black run. These are automatic; the developer sees the results immediately.
- **PostToolUse on Edit/Write (Code Gen)**: For some stacks, there's boilerplate that needs updating. Add a new route in Rails? The hook updates route registrations. Add a new GraphQL type? The hook generates resolvers. This keeps boilerplate in sync automatically.
- **PreCommit**: Before code reaches git, hooks run the full test suite and linting. Nothing commits if quality gates fail. This prevents broken code from reaching the repository.

Hooks are powerful but also dangerous if over-applied. The `/bootstrap` system is careful: it only creates hooks for genuinely frequent operations, not rare ones. Otherwise, they become noise.

## The /scope Conversation Design

The `/scope` conversation is structured but natural. Think of it as a senior architect interviewing a client to understand their needs.

**Phase 1: Vision**

"Tell me what you're building. What problem does it solve? Who's the user? Why does this matter?" The system listens, asks clarifying questions, and builds a clear picture of the problem domain. It's not asking for a feature list; it's asking for understanding.

**Phase 2: Technical Reality**

"Where does this run? What's the operating environment?" Is it a mobile app or a web app? Is it deployed on the cloud, on-premise, or both? Does it talk to existing systems? What's the data model at a high level? These answers constrain the architecture significantly.

**Phase 3: Constraints**

"What's the timeline? What's the team's experience level? Do you have any non-negotiable tool preferences or organizational standards?" These answers shape decisions about complexity, boilerplate, and risk.

**Phase 4: MVP**

"What's in version 1? What absolutely has to work for this to be valuable?" This produces the MVP definition that feeds into `PHASES.md`. It prevents over-scoping and keeps the first phase focused.

The output is `scope.json`, a structured file that captures all of this in a form that both humans and machines can work with. Developers can edit it, and the next generation pass will see the changes.

## The /bootstrap Generation System

Once `/scope` is complete, `/spec` generates the conceptual architecture, and now `/bootstrap` makes it concrete.

The generation process works in stages:

1. **Analysis**: Read the spec and architecture. Identify the tech stack, key patterns, and project structure. Recognize whether this is a service, a client, a library, a data pipeline, etc.

2. **Skill Generation**: For each artifact type the project needs (components, models, migrations, tests, endpoints, etc.), generate a skill that encodes the right way to create that artifact. The skill includes the common structure, the naming convention, the testing approach, and the integration points.

3. **Hook Generation**: Create hooks for the frequent operations. A JavaScript project gets ESLint validation. A Python project gets Black formatting and mypy checking. A Rust project gets clippy. These run automatically and keep code quality consistent.

4. **Rule Encoding**: Document the project's architectural rules in a machine-readable form. "State changes happen here, not there." "Async operations follow this pattern." "Error handling uses this strategy." These rules guide code generation and become checkpoints in `/phase-review`.

5. **Agent Creation**: For stack-specific reasoning tasks (code review, test suggestion, architecture validation), create agents that understand the project's patterns and can reason about them.

6. **CLAUDE.md**: Synthesize everything into a comprehensive human-readable guide. What's the tech stack? What's the folder structure? How do you add a new feature? What are the conventions? Where do things go? This becomes the reference guide for all future development.

The key is that none of this is template-filling. It's AI-authored reasoning. A Ruby on Rails project and a Django project both need to handle HTTP requests, but the implementation is completely different, and `/bootstrap` understands that. It generates appropriate artifacts, not generic ones.

## Context Management Strategy

Claude Code's context window is large but finite. Claude Forge includes strategies to use context intelligently.

**Session Handoff Pattern**

When a session ends, `/wrap-up` produces two artifacts:

1. **Human-readable handoff** (stored in CLAUDE.md): A summary of what was done, what's next, and any gotchas. Next developer reads this and understands the state.

2. **Machine-readable context** (stored in transfer-context.json): Structured data about which files changed, which tests passed, what was learned, what's incomplete. `/dev` in the next session reads this and re-orients quickly.

**Compaction Strategy**

When context utilization approaches 60%, the `/compact` hook triggers. Instead of losing context, it:

- Summarizes the architecture (one page, key decisions)
- Summarizes the current phase goals (what we're working on now)
- Keeps the most recent `/wrap-up` outputs (what was just done)
- Clears verbose operation logs and intermediate attempts

This preserves the signal and drops the noise. The next session loads with less context, but more relevant context.

**Parallel Worktrees**

For large projects where multiple features are in flight, Claude Forge suggests using Git worktrees. Feature A works in one worktree, Feature B in another. `/dev` runs against one worktree at a time, context stays focused, and merges happen cleanly.

**Subagent Delegation**

For verbose operations (like generating a large test suite or refactoring a big module), Claude Forge can spawn subagents. These work in separate context windows, producing outputs that the main agent integrates. This keeps the main context focused on orchestration while subagents handle depth.

## Self-Improvement Loop

Claude Forge is designed to get smarter over time. The system includes mechanisms for continuous improvement:

**Learning Capture**

Every significant discovery goes to `/learn`. "We discovered that this library doesn't work well with async initialization." "This approach is too slow for datasets over 10GB." These lessons are deduplicated, stored in a project knowledge base, and automatically incorporated into CLAUDE.md and decision guidance for future developers.

**Skill Auditing**

Generated skills can become stale. Maybe the project's patterns evolved. Maybe the underlying library changed. `/skill-check` runs regularly (or manually), tests each generated skill, and flags any that no longer match reality. It proposes updates or suggests replacing them with new skills that better reflect current patterns.

**Quality Feedback**

`/phase-review` produces quality scores and identifies gaps. These gaps feed back into the system. If code quality is consistently below 80/100 for a particular type of artifact, that signals a skill gap. Maybe we need better testing guidance, or the skill for that artifact type isn't aligned with reality.

**Architecture Evolution**

As projects grow, initial architecture decisions may not scale. `/bootstrap` can be re-run periodically (especially at phase boundaries). It re-analyzes the current state and can propose architectural improvements, new skills, or refactored hooks.

**Documentation Synchronization**

Every skill has inline documentation. Every hook has rationale. When these change, CLAUDE.md auto-updates to reflect reality. Developers don't consult stale docs; the system keeps documentation in sync with implementation.

## Quality Assurance

Quality isn't an afterthought; it's built into the system.

**PreCommit Gates**

Before code reaches the repository, `/wrap-up` ensures:

- Tests pass (100% of modified code has tests; coverage meets project minimum)
- Linting succeeds (no style violations)
- Architecture rules are respected (changes don't violate layering, dependency flow, etc.)
- Documentation is current (CLAUDE.md reflects changes)

Nothing commits if any gate fails.

**Phase-Review Scoring**

`/phase-review` scores work across five dimensions:

1. **Completeness**: Were planned features delivered? (0-20 points)
2. **Code Quality**: Does the code meet project standards? (0-20 points)
3. **Test Coverage**: Are changes adequately tested? (0-20 points)
4. **Architecture Alignment**: Does new code respect the architecture? (0-20 points)
5. **Documentation**: Is everything documented? Are CLAUDE.md and runbooks current? (0-20 points)

Total: 100 possible points. Phases don't ship below 80/100. If something's below that threshold, `/phase-review` identifies exactly what needs improvement and suggests fixes.

**Provocation Prompts**

During planning (`/dev`), the system includes "provocation" prompts. "Have we considered performance implications?" "What about error cases?" "Is this consistent with how we handle similar problems elsewhere?" These aren't blockers; they're thinking tools. Developers answer or dismiss them, but they ensure thinking is thorough before building.

**Guard Phrases**

To prevent skipping planning, `/dev` requires developers to make explicit design decisions before building. You can't just say "implement feature X." You have to articulate the approach, the risks, and the rationale. This sounds like friction, but it prevents costly rework later.

## Usage Guide

Here's how a developer uses Claude Forge from day one:

**Onboarding (First Session)**

```
# Clone the template
git clone https://github.com/you/claude-forge my-new-project
cd my-new-project

# Start Claude Code
claude

# Phase A: Genesis (20-30 minutes total)
/scope          # Interactive discovery (~15 min)
/spec           # Generate spec + architecture (~5 min)
/bootstrap      # Generate custom skills, hooks, rules (~5 min)

# Commit the bootstrap
git add .
git commit -m "Initial project bootstrap"
```

At this point, the developer has a complete development environment. No more setup needed.

**Daily Development (Every Session)**

```
# Start Claude Code (in existing project)
claude

# One or more development tasks
/dev "Add user authentication"       # Build a feature
/dev "Refactor payment processing"   # Refactor something
/dev "Write performance test"        # Test something

# End the session
/wrap-up        # Summarize, commit, hand off
```

Each `/dev` is a focused feature. The developer describes what they want to build, `/dev` orchestrates the plan-build-verify cycle, and the output is tested, documented, and ready for review.

**Periodic Maintenance**

```
# End of phase
/phase-review   # Audit work, score quality

# If quality is below 80
# /dev <fixes>  # Address gaps

# Periodic skill health check
/skill-check    # Audit skills, flag anything stale

# If something needs learning
/learn "We discovered X about our codebase"
```

**Regeneration (When Architecture Evolves)**

```
# Major architectural shift happens
# Update ARCHITECTURE.md manually, then:
/bootstrap      # Regenerate skills, hooks, rules

# Review changes, commit, move on
```

This is the full lifecycle. What used to take hours of setup now takes minutes. What used to be ad-hoc development becomes systematic and repeatable. Developers spend less time figuring out where things go and more time building.

---

## Conclusion

Claude Forge is a system for baking best practices into project templates. It recognizes that most software development follows predictable patterns: plan carefully, build systematically, verify thoroughly, learn continuously. The technology stack changes, but the cycle remains. By separating universal workflow from stack-specific implementation, Claude Forge makes it possible to bootstrap new projects with a fully configured, best-practice development system in a single session, then sustain productive development through consistent, AI-guided workflow.

The system gets smarter over time. Skills evolve. Patterns emerge and become codified. Documentation stays current. Quality gates keep the bar high. And every lesson learned becomes part of the project's institutional knowledge, available to every future developer.
